# GraphQL + NestJS cutover readiness

**Status:** Phase 0 baseline validated; Phase 1 contact read/write, bounded bulk-mutation, activity, and bounded related-content slices implemented; staging CRUD, bulk, activity, related-content, rollback, and operation-observation gates completed

**Evidence date:** 2026-07-17
**Authority:** This document supersedes the testing counts and cutover-testing guidance in `ts-nest-rewrite.md`.

## Decision

The target public API is GraphQL on NestJS. Do not rebuild the complete REST surface in NestJS as a prerequisite. Extract transport-neutral domain services, expose them through GraphQL, and retain the legacy Express routes until each consumer scenario has passed a semantic cutover gate. A temporary NestJS REST adapter is appropriate only when a specific client cannot move with its GraphQL operation.

Parity means the same authorization decisions, domain state transitions, side effects, and user-visible outcomes. It does not mean forcing GraphQL responses and errors to imitate REST envelopes byte-for-byte.

## Phase 1 GraphQL foundation now in place

The isolated `backend-v2/` workspace now boots NestJS 11 with code-first GraphQL and Apollo Server 5. It is not part of the legacy Railway build/start commands and receives no production traffic.

The first vertical foundation implements the existing access-cookie contract, verifies `itemize_auth` with the shared JWT secret, selects organization context only from `x-organization-id` or the user's database default, re-reads current membership and role from PostgreSQL, and propagates the verified identity through `AsyncLocalStorage`. A public `readiness` query and protected `viewerContext` query prove the middleware and guard chain.

The first domain slice exposes tenant-scoped `contacts` and `contact` queries plus single-row and bounded bulk contact mutations through a transport-neutral service and parameterized repository. Reads preserve legacy search, status, tag, assignee, sorting, and page semantics while adding strict inputs and deterministic ID tie-breaking. Writes serialize plan-limit checks, verify assignee membership, use organization-qualified locks and writes, and commit contact-created/contact-updated/tag workflow triggers and activities atomically with domain state. Updates preserve omitted fields and clear nullable fields only when null or empty is explicit. Bulk requests are capped at 100 IDs, deduplicated in request order, mutate only matched active-organization rows, and return requested/matched/changed/rejected sets without revealing whether a rejected ID is foreign or absent. Cross-tenant single-row reads and mutations return tenant-private `NOT_FOUND`. The comparison suite also found and repaired ambiguous unqualified columns in the legacy contact search query.

`npm run build:graphql` passes. `npm run test:graphql` covers 49 focused cases across the foundation, credentialed CORS allowlisting/preflight, organization context, CSRF missing-cookie/header/mismatch/success behavior, contact, bulk, activity, and related-content validation/error mapping, repository SQL/transaction behavior, and operation observability. The disposable runner executes 22 NestJS GraphQL cases against freshly initialized PostgreSQL: 4 context cases plus 18 contact cases covering dual REST/GraphQL reads, create normalization and side effects, concurrent last-slot plan enforcement, omitted-versus-null updates, changed-field trigger idempotence, tenant-private mutation misses, assignee denial, CSRF rejection, exact single deletion, bounded mixed-tenant bulk update/delete, no-op retry idempotence, atomic assignment denial, workflow/activity side effects, dependent-row cleanup, structured activity creation, activity filter parity/privacy, and bounded related-content parity/privacy. The shared frontend contact API uses independent `VITE_CONTACT_READS_GRAPHQL`, `VITE_CONTACT_MUTATIONS_GRAPHQL`, `VITE_CONTACT_BULK_MUTATIONS_GRAPHQL`, `VITE_CONTACT_ACTIVITIES_GRAPHQL`, and `VITE_CONTACT_CONTENT_GRAPHQL` flags, maps results back to the existing consumer shape, includes credentials and verified-organization selection, obtains the retained HTTP CSRF token for writes, recovers once from GraphQL `UNAUTHENTICATED`, and defaults every operation to REST for immediate rollback. Related-content collections expose total and truncation state at a 100-row bound; the current adapter fails closed when pagination UI would be required rather than hiding rows. The legacy-origin proxy forwards the CSRF header without broadening its existing header allowlist. The NestJS service accepts credentialed browser requests only from `FRONTEND_URL`, fixed production origins, or explicit `EXTRA_CORS_ORIGINS`; local Vite origins are development-only. Privacy-safe proxy and NestJS events now correlate the request ID and expose transport/layer, operation name/type, numeric latency, operation/error counters, and stable error codes without GraphQL source, variables, response data, or identity fields. All five flags remain off in deployed frontend builds and no production traffic reaches NestJS.

## Per-slice exit gate

A slice is ready to cut over only when all of the following are true:

- its ledger row has an explicit disposition and owner
- auth and tenant-denial cases pass
- success, validation, not-found, conflict, and failure paths are characterized
- database state and external side effects match the intended semantics
- retries are safe for mutations that can be repeated
- the frontend or external consumer uses the new operation successfully
- metrics/logs distinguish legacy and GraphQL execution
- rollback does not require data repair

The Express backend can be retired only after all externally used ledger rows are migrated, deliberately retained, or deliberately removed and the observation window shows no legacy traffic.
